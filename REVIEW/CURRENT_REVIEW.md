# 当前版本问题反馈与优化方案

## 结论

当前版本相比初版已经有明显进步：项目从一个单脚本 MVP，升级成了有统一引擎、缓存兜底、配置校验、回测和结构化 AI 风险旗标的个人 ETF 决策工具雏形。

整体判断：

- 工程结构：明显改善。
- 周报可运行性：明显改善。
- 投资策略验证：已有基础回测，但还不充分。
- 个人使用价值：适合作为 ETF 资产配置和决策纪律工具。
- 当前定位：不应被理解为“能稳定赚钱的量化系统”，更适合作为“个人资产配置驾驶舱”。

当前评分建议从上版的 `5.5 / 10` 提升到 `7 / 10`。

## 已完成的关键改进

### 1. 单一事实源已经建立

当前实现集中在 `engine/`：

- `engine/signals.py`
- `engine/backtest.py`
- `engine/validate_flags.py`
- `engine/flags_schema.json`

`.claude/skills/weekly-briefing/SKILL.md` 和 `.agents/skills/weekly-briefing/SKILL.md` 都变成薄入口，只负责调用同一套 `engine/` 实现。这解决了之前路径不一致、两套脚本可能漂移的问题。

### 2. 数据缓存和数据质量分级已经加入

`engine/signals.py` 已经支持：

- 东方财富数据源。
- 新浪备用源。
- 本地缓存兜底。
- 数据质量分级：完整 / 缓存可用 / 过旧 / 部分缺失。
- `rebalance_allowed` 控制是否允许输出再平衡建议。

实际运行结果显示，即使外部估值接口不可用，行情仍可通过缓存产出，当前行情最旧约 1 天。

### 3. 配置校验已经有基础能力

当前已经校验：

- `portfolio.yaml` 是否有 holdings。
- `cash` 是否为非负数字。
- `shares` 是否为非负数字。
- `target_weight` 是否在 0 到 1。
- `target_weight` 合计是否接近 1.0。
- ETF 代码是否重复。
- ETF 是否存在于 `strategy.yaml` 的 universe。

这能减少手工维护 portfolio 时的低级错误。

### 4. 回测已经可运行

`engine/backtest.py` 可以离线使用缓存数据运行，当前结果显示：

| 组合 | 年化 | 最大回撤 | 结论 |
|---|---:|---:|---|
| 本策略：趋势过滤 | 约 4.2% | 约 -10.2% | 降低回撤，但牺牲收益 |
| 静态组合：无趋势过滤 | 约 5.9% | 约 -16.0% | 收益更高，但回撤更大 |
| 沪深300买入持有 | 约 4.4% | 约 -45.1% | 波动和回撤显著更高 |

这说明项目已经能回答一部分核心问题：这套 ETF 配置本身有价值，但趋势过滤并不一定提高收益。

### 5. AI 舆情层开始规则化

`engine/flags_schema.json` 和 `engine/validate_flags.py` 已经把 AI 风险旗标变成可校验结构，避免 AI 随意发挥。

这比“AI 搜新闻自由总结”更可靠。

## 当前主要问题

### P1：回测样本期偏短，不能支撑长期结论

当前回测区间约为：

```text
2020-02-05 -> 2026-06-02
```

约 6.1 年。这个区间能说明近期表现，但不足以支撑长期年化 5% 的结论。

原因：

- 样本覆盖的完整市场周期不够多。
- 起点受 ETF 成立时间限制，尤其 `512890` 这类较新的 ETF 会缩短整体交集。
- 使用 ETF 实盘历史会受基金成立时间、分红复权、流动性变化影响。

优化方向：

- 保留 ETF 回测作为真实可交易验证。
- 增加指数代理回测，用指数延长历史区间。
- 明确区分“ETF 实盘可交易区间”和“指数代理长期区间”。
- 在 README 和回测输出中标注样本期限制。

### P1：估值数据没有缓存，网络失败时会静默缺失

当前行情已经有缓存，但估值仍直接依赖外部接口。外部接口失败时，`signals.json` 只是不写 `valuation` 字段。

这会带来两个问题：

- 周报模板要求展示估值分位，但数据可能缺失。
- AI 层可能把“没有估值字段”误读成“估值中性”。

优化方向：

- 给估值增加缓存，例如：

```text
engine/cache/valuation_沪深300.json
engine/cache/valuation_中证500.json
```

- 在 `signals.json` 增加估值状态：

```json
{
  "valuation_status": {
    "510300": {
      "available": false,
      "source": null,
      "reason": "network_failed"
    }
  }
}
```

- 网络失败时优先使用未过旧缓存。
- 缓存过旧时明确标注，不参与判断。

### P1：趋势过滤策略目前未证明优于静态组合

回测结果显示：

- 趋势过滤策略年化约 4.2%。
- 静态组合年化约 5.9%。
- 趋势过滤最大回撤更低。

这说明趋势过滤更像风险控制工具，而不是收益增强工具。

优化方向：

- README 和周报中避免暗示趋势过滤可以提高收益。
- 把趋势过滤定位为“降回撤机制”。
- 增加风险偏好选项：
  - 保守：启用趋势过滤。
  - 平衡：仅提示趋势，不自动迁移仓位。
  - 进取：主要使用静态再平衡。

### P2：`signals.json` 缺组合级行情日期字段

当前每只 ETF 都有 `as_of`，但顶层没有组合级 `as_of`。

两个入口模板都要求展示：

```text
行情截至 <as_of>
```

如果不同 ETF 日期不同，单个 `<as_of>` 不够准确。

优化方向：

在 `signals.json` 顶层增加：

```json
{
  "as_of_min": "2026-06-02",
  "as_of_max": "2026-06-03",
  "as_of_summary": "2026-06-02 至 2026-06-03"
}
```

周报展示时使用 `as_of_summary`。

### P2：回测缓存缺少元数据

当前 `engine/data/*.csv` 只保存价格，没有记录来源和复权状态。

这会影响后续判断：

- 数据来自东方财富还是新浪。
- 是否前复权。
- 何时拉取。
- 缓存是否需要刷新。
- 回测结果是否包含未复权偏差。

优化方向：

增加：

```text
engine/data/meta.json
```

记录每只 ETF：

```json
{
  "510300": {
    "source": "eastmoney",
    "adjust": "qfq",
    "fetched_at": "2026-06-03T15:00:00+08:00",
    "start": "2012-05-28",
    "end": "2026-06-02",
    "rows": 3400
  }
}
```

回测输出中引用这些 metadata，而不是笼统说“缓存”。

### P2：`strategy.yaml` 自身校验不够

当前配置校验主要覆盖 `portfolio.yaml`。如果 `strategy.yaml` 因子参数缺失或非法，脚本可能在后续阶段报 KeyError，或者生成不合理结果。

优化方向：

增加 strategy 校验：

- universe 代码不能重复。
- universe 中必须有且只有一个 `asset: bond`。
- `ma_days > 0`。
- `lookback_days > 0`。
- `lookback_years > 0`。
- `cheap_pct < rich_pct`。
- `0 <= cheap_pct <= 1`。
- `0 <= rich_pct <= 1`。
- `abs_threshold_pp > 0`。
- `0 < rel_threshold <= 1`。
- 所有 `enabled` 字段必须是 bool。

### P2：当前 portfolio 仍是空仓模板

`portfolio.yaml` 里：

```yaml
cash: 0
shares: 0
```

这意味着当前 live 周报不能验证真实再平衡建议，只能验证信号链路。

优化方向：

- 用真实持仓填一次。
- 或新增 `examples/portfolio.example.yaml` 作为模板。
- 根目录 `portfolio.yaml` 保留真实账户数据，但注意不要提交敏感信息。

### P3：`engine/validate_flags.py` 对缺失 flags.json 的体验偏硬

当前如果没有 `engine/flags.json`，校验器直接报错。按技能流程，这是合理的，因为 AI 应该先写 flags 文件。

但从个人使用体验看，也可以更温和：

- 提供 `--init-empty` 参数生成空 flags。
- 或在无文件时提示一条可复制的最小模板。

优化方向：

```bash
python3 engine/validate_flags.py --init-empty
```

生成：

```json
{
  "generated_for": "2026-06-03",
  "flags": []
}
```

### P3：README 对“已完成”的表达可以更审慎

README 写了“本项目有回测”，这是事实。但“已完成：回测 + 参数敏感性 + 成本影响”容易让人误解为策略已经充分验证。

优化方向：

把表达改成：

```text
已完成基础回测：可用于初步验证，但仍需更长历史、指数代理和更多市场周期测试。
```

## 推荐优化优先级

### 第一优先级：提高周报可信度

1. 增加组合级 `as_of_min` / `as_of_max` / `as_of_summary`。
2. 增加估值缓存。
3. 增加估值缺失状态。
4. 周报中明确“估值缺失不等于中性”。

原因：这些直接影响每周报告是否会误导用户。

### 第二优先级：提高回测可信度

1. 给回测数据增加 metadata。
2. 在回测输出中自动给出策略判断。
3. 增加指数代理长期回测。
4. 明确区分 ETF 可交易回测和指数代理回测。

原因：这是判断项目是否有长期投资价值的核心。

### 第三优先级：提高配置安全性

1. 强化 `strategy.yaml` 校验。
2. 增加示例 portfolio。
3. 防止真实持仓误提交。

原因：配置错误是个人工具最常见的风险源。

### 第四优先级：增强个人投资系统能力

1. 增加风险预算。
2. 增加动作门槛。
3. 增加交易后复盘。
4. 增加每周决策日志。

原因：这些能把工具从“信号生成器”升级成“可复盘的个人决策系统”。

## 建议实施顺序

### 第 1 批：小改动，高收益

- `signals.py` 增加组合级行情日期字段。
- `signals.py` 增加估值缺失字段。
- SKILL.md 模板使用 `as_of_summary`。
- README 调整回测表述。

### 第 2 批：数据可靠性

- live 估值缓存。
- 回测数据 metadata。
- 回测输出显示数据来源和复权状态。

### 第 3 批：策略可信度

- strategy 配置校验。
- 回测结论自动摘要。
- 指数代理长期回测。

### 第 4 批：个人化能力

- 风险预算配置。
- 动作门槛。
- 交易后复盘日志。

## 当前版本的合理定位

当前项目适合这样使用：

> 每周帮助用户检查 ETF 组合状态、行情趋势、动量强弱、再平衡偏离和外部风险，辅助用户做低频、克制、可复盘的资产配置决策。

当前项目不适合这样使用：

> 自动判断市场涨跌、推荐个股、承诺年化收益、替用户直接下单。

最值得坚持的边界是：

- 只做 ETF，不做个股推荐。
- 只做低频周报，不做高频交易。
- AI 只做解释、检查和风险旗标，不做最终拍板。
- 输出动作必须可追溯到规则和数据。

## 总体评价

这版已经值得继续投入。它不是专业量化平台的替代品，但作为个人私有化投资工具，有明确价值：

- 它能服务个人真实约束。
- 它能沉淀决策记录。
- 它能减少随意交易。
- 它能让 AI 的作用被限制在更安全的位置。

下一阶段不建议优先做网页、推送或复杂 AI 自动化。更好的方向是继续补可信度：

1. 数据日期更清楚。
2. 估值缺失更透明。
3. 回测口径更严谨。
4. 风险预算更个人化。
